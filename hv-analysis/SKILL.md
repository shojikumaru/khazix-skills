# 縦横分析法による深度研究

> **方法論の来歴**
> 縦横分析法は Khazix（数字生命卡兹克）が提唱したフレームワーク。言語学の通時・共時分析（Saussure）、社会科学の縦断・横断研究デザイン、ビジネススクールのケーススタディ法、そして競争戦略分析のエッセンスを取り込み、プロダクト／企業／概念／人物を対象に使える汎用研究フレームとしてまとめ直したもの。根本原則は変わらない：**縦軸で時間の深さを追い、横軸で同時点の広がりを追い、最後にその交点から判断を導く**。

あなたは今、縦横分析法による深度研究を1件実行している。最終成果物は **装丁済みの PDF 研究レポート**。

## 事前準備

### 環境準備

1. **PDF 変換スクリプトの用意**：本 Skill には `scripts/md_to_pdf.py`（WeasyPrint ベース）を同梱している。最終 Markdown レポートを装丁済み PDF に変換するためのもの。依存が未インストールなら：`pip install weasyprint markdown --break-system-packages`。
2. **文体ガイド**：本 Skill に文体ガイドを完全内蔵済み（下の「文体」セクションを参照）。別 skill を追加で読み込む必要はない。

### 研究対象の明確化

ユーザー入力を受け取ったら、以下を確認する。ただし既に十分明確（例：「縦横分析法で Hermes Agent を研究して」）なら聞き返さず、そのまま開始してよい。

1. **研究対象**：具体的なプロダクト名／企業名／概念名／人物名
2. **種別**：プロダクト、企業、概念、人物、その他のどれか？
3. **研究動機**（任意）：なぜそれを研究したいか？直近で何が起きたか？
4. **特に深掘りしたい観点**（任意）：踏み込みたい方向性はあるか？

## ステップ1：ネット経由の情報収集

この方法論の質は、情報の豊かさと正確さによって完全に決まる。**必ずネット検索する**こと。既存知識だけで済ませてはいけない。研究レポートの価値は深さと完全性にあるので、情報収集段階は「搜りすぎるくらいがちょうどいい」。ここで手を抜くと後段の分析が表層的になる。

### 並列検索戦略

効率化のため、サブエージェントを並列で走らせる。推奨の役割分担：

- **サブエージェント1 — 縦軸情報**：研究対象の起源、創業者・提唱者の背景、発展経緯、重要イベント、バージョン遍歴、資金調達、戦略転換、危機や炎上
- **サブエージェント2 — 横軸情報**：競合の洗い出し、各競合の特徴とユーザー評判、業界内の比較評論、市場シェア
- **サブエージェント3**（複雑な対象の場合のみ）：補足情報。創業者の深い経歴、業界環境の変化、ユーザーコミュニティの議論（GitHub Issues、Reddit、Twitter/X、Qiita、Zenn、はてな、知乎 など）

**サブエージェントへのネット利用ガイド**（各サブエージェントの prompt に直接書き込むこと）：

各サブエージェントの prompt には必ず以下のネット利用指針を含める：

> あなたはネット経由で情報を取得する必要がある。使用ツール：
> - **WebSearch**：情報源を発見し、要約やキーワード結果を得るための検索に使う
> - **WebFetch**：具体的な URL が判明している場合、ページからピンポイントに内容を抽出するのに使う
> - もしユーザー環境に web-access skill がインストールされていれば（`/mnt/.claude/skills/web-access/SKILL.md` の存在を確認）、それを優先的にロードして指針に従うこと。ブラウザ CDP によるより強力な取得能力を提供する
> - 検索戦略：まず WebSearch で情報源の手がかりを掴み、具体 URL を得たら WebFetch で深掘り抽出する
> - 複数回、複数のキーワード組み合わせで検索すること。1 回で諦めない
> - **一次情報 > 二次情報**：公式ブログ > 権威媒体のオリジナル報道 > 転載・アグリゲーター
> - **学術的対象は arxiv 必須**：研究対象が学術概念、アルゴリズム、AI モデル、技術パラダイムなどを含むなら、必ず arxiv API で関連論文を取得すること。呼び出し例：`curl -s "https://export.arxiv.org/api/query?search_query=all:キーワード1+AND+all:キーワード2&max_results=10"`、または WebFetch で同 URL を取得。XML 形式でタイトル、著者、アブスト、公開日、PDF リンクが返る。キーワード組み合わせや件数は必要に応じて調整。重要論文が見つかったら WebFetch で論文ページ（`https://arxiv.org/abs/論文ID`）を読み込み、追加の詳細を取得する。

prompt には目的語（「取得する」「調査する」「把握する」）を書き、具体手段を匂わす動詞（「検索する」「クロールする」）は避ける。サブエージェントに最適な取得手段を自分で判断させる。

### 情報源の優先順位

一次情報は二次情報に勝る。複数メディアが同じ誤りを引用すると「循環引用による疑似的な裏付け」が生まれるので注意する。

| 情報の種類 | 一次情報 |
|---|---|
| プロダクト更新／技術決定 | 公式ブログ、GitHub Release Notes、創業者の SNS 投稿 |
| 資金調達／ビジネスデータ | 企業公式発表、SEC・法人登記書類 |
| ユーザー評判 | GitHub Issues、Reddit スレッド、Twitter/X、はてブ、知乎 |
| 業界分析 | 権威媒体のオリジナル報道（転載ではないもの） |
| 学術／技術原理 | arXiv 論文（`export.arxiv.org/api/query`）、Google Scholar、学術会議論文集 |

### 情報充足度のセルフチェック

検索が終わったら以下を確認する：

- 縦軸：一続きの物語として語れるか？明らかな情報欠落はないか？
- 横軸：競合リストは網羅的か？主要プレイヤーを取りこぼしていないか？各競合について比較可能なだけの情報があるか？
- 出典：重要事実に信頼できる出典があるか？単一ソース依存のまま判断を下していないか？

情報が足りなければ、追加で掘る。妥協しない。

## ステップ2：縦軸分析（Diachronic / Longitudinal）

時間軸に沿って、研究対象が誕生から現在に至るまでの発展全貌を余すところなく再構築する。レポートの主体部分であり、分量も最も厚くなる。

### 内容要件

**起源の追跡**：それが生まれた背景は何か？どの技術・思想・ニーズの上に立って現れたか？創業チームや中心的推進者は誰か？彼らは以前どんなことをしていて、なぜ彼らがこれをやることになったか？当時の業界環境はどうだったか？誕生を直接促した鍵となるイベントや閃きがあったか？

**誕生時点**：最初のリリース／設立／提唱の時期を特定する。当初の形と位置付け、そして現在との違い。

**発展の経緯**：誕生から現在まで、時系列に主要なノードをすべて洗い出す。含むべきもの（例示、網羅ではない）：大きなバージョン更新、資金調達、チーム変動、戦略転換、技術アーキテクチャの変化、ユーザー規模のマイルストーン、重要提携・買収、広報危機や論争。

**意思決定のロジック**：各ノードで、背後にあった意思決定の理由をできる限り復元する。なぜ A ではなく B を選んだか？当時の制約条件は何だったか？どの初期決定がその後の方向性を「固定化」し、後戻り不能にしたか？どういう機序で深みにはまっていったか（ネットワーク効果、エコシステムの絡みつき、技術スタック選択など）？

**フェーズ分割**：全歴程を自然な単位（黎明期、急成長期、転換期、成熟期 など）でフェーズに分ける。各フェーズには中核的特徴と中核的矛盾がある。

### 分量

6,000〜15,000 字。歴史が長くノードが多い対象は上限寄り、新しい対象は下限寄り。核は「物語を完結させ、掘り切る」こと。各ノードは展開に値する。圧縮のために重要ディテールを飛ばしてはいけない。長く細かく書く方が、薄く流すよりよほどいい。

## ステップ3：横軸分析（Synchronic / Cross-sectional）

現在時点を断面として、研究対象を同じ戦場の競合／同類と網羅的に比較する。

### まず競合状況を判定する

3 つのシナリオに分けて処理する：

**シナリオ A：直接的な競合なし。** 研究対象が完全な新カテゴリ、または独占性が極めて強い領域にある場合、逐一比較はスキップし、代わりに次を分析する：なぜ競合が現れないのか？カテゴリが新しすぎるのか、参入障壁が高いのか、市場が小さすぎるのか？将来、競合はどの方向から出てくる可能性が高いか？間接的な代替手段、あるいは前世代の解決策で参照できるものはあるか？

**シナリオ B：競合が少数（1〜2 個）。** それぞれを深掘り比較し、競合ごとに詳細分析を展開する。

**シナリオ C：競合が充分にある（3 個以上）。** 最も代表的な 3〜5 個を選んで比較し、残りは簡潔に触れる。

### 比較軸

研究対象の種別に合わせて柔軟に調整するが、少なくとも以下を網羅する：

**中核的な差異比較**：技術ロードマップ／中核方法論／基盤ロジック、プロダクト形態／ビジネスモデル／組織構造、ターゲットユーザー／対象読者／想定シナリオ、中核的強みと明らかな短所、価格戦略／投入リソース／規模感。

**ユーザー視点**：各競合に対する実ユーザーの評判はどうか？コミュニティの評価や使用体験でもっとも言及される長所と不満はそれぞれ何か？ユーザーの実利用方法と公式ポジショニングの間にズレはあるか？比較は「スペック対照表の文字版」に堕させず、各競合が「実際にどう生きているか」、ユーザーがそれを選ぶ真の理由を語ること。

**ニッチ分析**：この戦場全体の版図の中で、研究対象はどの位置を占めているか？どの空白を埋めているのか、あるいは誰と正面から競っているのか？現在の構図は百花繚乱か、二強対決か、一強独走か？

**趨勢の見立て**：横比較から見て、研究対象は競争構図の中でどの方向に向かっているか？機会とリスクはそれぞれ何か？

### 分量

3,000〜10,000 字。シナリオ A は 3,000 字前後に収め、シナリオ C では主要競合 1 つあたり少なくとも 1,500 字以上の独立分析を展開する。流し読みで済ませない。

## ステップ4：縦横交差の洞察

レポート全体の精髄セクション。縦軸の発展経緯と横軸の競争構図を結合して、総合的かつ新しい判断を出す。前のセクションの縮約版になってはいけない。

答えるべき核心の問い：

1. **歴史が現在の競争ポジションをどう形作ったか**：縦軸の経緯にあった決定やイベントのうち、どれが今日の横軸上のポジションを決めたのか？
2. **競合の縦軸比較**：主要競合も時間軸に乗せて見たとき、起源と進化経路はどう違うか？その違いがどう今日の各社の特徴を生んだか？
3. **強みの歴史的ルーツ**：今日の中核的強み 1 つ 1 つは、歴史上のどのノード／決定に辿れるか？
4. **弱みの歴史的ルーツ**：今日の中核的弱みは、過去のどの決定に辿れるか？当時の「良い決定」が、今では重荷になっていないか？
5. **未来シナリオ**：縦軸の趨勢と横軸の競争構図を踏まえ、3 つの筋書きを提示する — **最も起こりそうな筋書き**、**最も危険な筋書き**、**最も楽観的な筋書き**。各筋書きに論理的な裏付けを伴わせる。

### 分量

1,500〜3,000 字。

## 文体

これは冷たいコンサルレポートではなく、**頭から最後まで読み通せる深度研究記事**である。文体は「研究レポートの厳密さ」と「Khazix 流の可読性」の間でバランスを取る。

### Khazix 文体から取り込む要素

以下の文体要素はレポート執筆にそのまま適用する（詳細定義は khazix-writer skill を参照）：

**リズム感**：文は長短を織り交ぜ、段落間にも自然な跳躍を作る。毎段落が同じ長さにならないように。1 文だけで段落を構成し、重みを持たせるテクニックも使ってよい。良いリズムは波のようで、本線から少し外れて、最後に「本線に戻す 1 文」で引き戻す、の繰り返し。

**物語駆動、列挙駆動ではない**：縦軸部分は物語の弧（起承転結）を持つこと。なぜこのプロダクトがある時点で突然ブレイクしたのか、伏線は何で、転機は何か、を描く。「2023 年 1 月に A をリリース、2023 年 3 月に B をリリース」というような年表の垂れ流しにしない。

**知識は「話しているうちにふと取り出す」もの**：叙述の流れの中で自然に背景知識を出す。「では、ここで◯◯について解説しましょう」のような教科書的な区切りは使わない。

**判断を下す勇気を持つ**：視点や洞察を出すことを推奨する。ただしすべての視点には事実による裏付けが必要。先に事実を置いてから判断する。推測はそれとはっきり明示する。判断を述べるときは「私はそう思う」「私の見立てはこうだ」のように主観性を認める姿勢で書き、見下ろすような断定はしない。

**層をめくる修辞**：いきなり結論を述べず、「現象 → 表面的な説明 → より深い問い → 中核的洞察」の順で展開する。読者を思考プロセスに参加させる。

**文化的次元の引き上げ**：交差洞察セクションでは、より大きな文化・哲学・歴史的参照点に接続する。無理な昇華ではなく、「話している流れで自然にそこに辿り着いた」感じで。

**呼応と回収**：冒頭や縦軸部分で仕込んだディテールやフックを、交差洞察やラストで呼び戻す（callback する）。前後の因果が閉じる感覚こそ、レポートを「情報の流れ」から「作品」に変える鍵。

### Khazix 文体から取り込まない要素

以下の要素は公众号記事（中国版 WeChat 公式アカウント向け長文記事）には合っても、研究レポートには合わない。抑制すること：

- **強すぎる口語化**：レポートに会話感があってよいが、「こいつマジで」「兄さん無理でしょ」「ヤバすぎ」みたいなスラングで紙面が埋まるのは NG。スパイス程度に使うのはよいが、密度は公众号記事よりはるかに低く。
- **見出し排除**：公众号記事は見出しを使わず一気に読み通させる作り。研究レポートは違う。1〜3 万字の内容に明確な構造とナビゲーションがなければ、読者は迷子になる。レポートには明確な章構造が要る。
- **記号の禁令は緩めてよい**：公众号記事ではコロンやダッシュを避けるが、研究レポートでは通常通り使ってよい。伝達効率が優先されるため。ただし「」によるカギ括弧の使い方は引き継いでよい。
- **固定尾部**：公众号の「いいね／保存／フォロー」的なテンプレ尾部は付けない。

### 絶対に避ける NG 表現（どの文体でも適用）

以下の「AI っぽさの指標」は、文体に関係なく避ける：

- 定型句：「まず…次に…最後に」「総括すると」「注目すべきは」「一見して分かるように」
- 中身のない形容詞：「エンパワーメント」「DX の打ち手」「閉ループを構築」「シナジー」
- 教科書的な出だし：「AI が急速に発展する今日において」「技術の進歩とともに」
- 多用しがちな地雷ワード：「ぶっちゃけ」「それが意味するのは？」「それが意味するのは」「本質的には」「言い換えれば」「否定できない」
- 具体名を避ける抽象ワード：「AI ツール」「あるモデル」ではなく、具体名を書く
- 捏造シーン：ある情報が見つからなかったら「該当情報は現時点で未確認」と正直に明示する。絶対に作らない。

※ この NG リストは中国語圏の文脈を日本語に翻案したもの。翔さん個別の文体ルール（特有の禁句、好む言い回し）がある場合はここを上書きすること。

### 人間の言葉で書く

コンサル会社風の定型句や抽象的な総括を避ける。「その企業はこの期間に急成長を実現した」ではなく、「2024 年半ばの ARR 1,000 万ドルから 2025 年末の 10 億ドルへ、成長曲線はほぼ垂直だった」のように、具体的なディテールと事例で置き換える。

## ステップ5：PDF レポートの生成

レポートを書き終えたら、本 Skill 同梱の `scripts/md_to_pdf.py` を使って Markdown を装丁済み PDF に変換する。

### 変換フロー

1. **まず Markdown 原稿を仕上げる**：完全なレポートを標準 Markdown 形式で書き、`[研究対象]_縦横分析レポート.md` として保存
2. **依存のインストール**（未インストールなら）：`pip install weasyprint markdown --break-system-packages`
3. **変換スクリプトを実行**：
   ```bash
   python [skillディレクトリ]/scripts/md_to_pdf.py input.md output.pdf --title "研究対象名" --author "著者名"
   ```
4. スクリプトは中間 HTML（デバッグ用）と最終 PDF の双方を自動生成する

### スクリプトに内蔵された装丁仕様

`md_to_pdf.py` は完全な CSS 装丁案を内蔵しているため、手動調整は不要：

- **ページ**：A4、マージン 上 25mm／左右 20mm／下 20mm
- **表紙ページ**：自動生成。タイトル（28pt 深青）、サブタイトル「縦横分析法 深度研究レポート」、著者情報、装飾線を含む
- **配色**：H1 見出し=#1a5276 深青、H2=#1e8449 緑、H3=#2e86c1 水色、H4=#5b2c6f 紫、本文=#2c3e50 濃灰
- **フォント**：CSS フォールバックチェーン `"Hiragino Sans", "Hiragino Kaku Gothic ProN", "Noto Sans CJK JP", "Noto Sans JP", Helvetica, Arial, sans-serif`（日本語主体、macOS/Linux 両対応）
- **本文**：10.5pt、行間 1.75、両端揃え、孤行・寡行の制御あり
- **引用ブロック**：左 3pt 深青の縦線 + 薄灰背景
- **表**：全幅、深青ヘッダー白文字、ゼブラストライプ行
- **ヘッダー**：「レポートタイトル | 縦横分析法 深度研究レポート」（表紙ページでは非表示）
- **フッター**：「p.X」のページ番号（表紙ページでは非表示）
- Markdown の最初の H1 は自動で表紙タイトルに抽出され、本文には重複表示されない

### Markdown 執筆時の注意

スクリプトに正しく解析させ、PDF 出力を最適にするため：

- 1 行目を `# タイトル` でレポートタイトルにする（表紙に自動転用される）
- タイトル直下に `> 研究日：... | 分野：... | 対象種別：...` の形式でメタ情報行を書くと、表紙に抽出される
- `##` を主要章の見出しに（縦軸分析、横軸分析、縦横交差 など）
- `###` と `####` を子章見出しに
- 表は標準 Markdown 表記法
- 引用は `>` 記法
- 太字は `**テキスト**`

### 末尾に入れる内容

Markdown 原稿の末尾に以下を追加する：
- **出典一覧**：引用した全ソースのリスト。URL とアクセス日時を明記
- **方法論について**：縦横分析法の由来に関する簡潔な注記（1〜2 文で十分）

## 対象種別ごとの適用アレンジ

根本原則（縦軸で時間の深さ、横軸で同時点の広がり）は不変だが、重心は対象種別によって変わる：

**プロダクトを研究するとき**：縦軸はバージョン遍歴、技術ロードマップの変遷、ユーザー増加カーブ、重要なプロダクト意思決定に重心を置く。横軸は機能比較、性能比較、ユーザー体験、価格に重心を置く。

**企業を研究するとき**：縦軸は創業チーム、資金調達史、戦略転換、組織変革、重要な人事変動に重心。横軸はビジネスモデルの差、市場シェア、売上比較、組織構造の差に重心。

**概念を研究するとき**（技術パラダイム、ビジネスモデル、文化現象）：縦軸はその概念の起源（誰が提唱したか、どの理論／ニーズに基づくか）、どう流行し、どんな論争と進化を経てきたかに重心。横軸は類似概念との違い、それぞれの適用シナリオ、各陣営の主張に重心。

**人物を研究するとき**：縦軸は個人の経歴、キャリア軌跡、重要意思決定、成長曲線、公開発言の変化に重心。横軸は同領域の他人物との対比（やり方、スタイル、実績、影響力、ルート選択の違い）に重心。

## 分量の全体感

| 部分 | 字数範囲 | 備考 |
|---|---|---|
| 縦軸分析 | 6,000〜15,000 字 | レポート本体、流し書きしない |
| 横軸分析 | 3,000〜10,000 字 | 競合数に応じて調整 |
| 縦横交差 | 1,500〜3,000 字 | 精髄セクション、新しい判断を出す |
| **全文総計** | **10,000〜30,000 字** | 長さを恐れない。深さと完全性こそ価値 |
